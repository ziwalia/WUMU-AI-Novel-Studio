# 改造方案 — 改前 vs 改后对比分析

---

## 改动 1：取消卷数，去掉"分卷"概念

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **用户操作** | 项目信息页需要手动设置"卷数"（1-20） | 不设置卷数，只设总章数 |
| **向导步骤** | 9 步（含"分卷大纲"） | 8 步（"分卷大纲"改为"小说大纲"） |
| **结构划分** | 按卷划分，每卷有章节范围、关键事件、角色变化 | 按幕/阶段划分，全书一个整体大纲 |
| **数据流** | 架构 → 分卷大纲 → 章节目录 | 架构 → 小说大纲 → 章节目录 |

| 优点 | 缺点 |
|:---|:---|
| 用户少填一个参数，流程更简洁 | 长篇小说（100章+）失去"卷"的天然分块结构 |
| 大部分网文本身不分卷，更贴合实际 | 一次性生成全书大纲，500章时后期章节质量可能下降 |
| 减少一个步骤，全自动生成更快 | 已有项目的 volumeOutline 数据需要迁移处理 |

> **建议**：取消卷数 UI 没问题，但 `novelOutlinePrompt` 内部应保留"阶段划分"逻辑。当章节数>50 时，自动要求 LLM 将全书划分为3-5个"大阶段"，每个阶段有独立的情感基调和冲突层次。这样简化了操作，又不丢失结构感。

## 改动 2：分卷大纲 → 小说大纲 + 蓝图去重

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **步骤名称** | "分卷大纲" | "小说大纲" |
| **输入** | 架构 + 卷数 + 角色/关系 | 架构 + 角色/关系（无卷数） |
| **输出内容** | 每卷：卷名、主题、章节范围、关键事件、角色变化 | 全书结构 + 各阶段：主题、章节范围、关键事件、情感基调曲线、角色变化 |
| **蓝图去重** | 无 | 目录生成后自动调用 LLM 分析章节结构相似度，标记雷同章节并给出替换建议 |

| 优点 | 缺点 |
|:---|:---|
| "情感基调曲线"是新增亮点，帮助后续章节保持节奏变化 | 去重校验多消耗1次 LLM 调用 |
| 从源头发现结构重复，比生成完再发现成本低得多 | LLM 的去重建议不一定准确，需要用户人工确认 |
| 用户可以在蓝图阶段就看到哪些章节可能雷同 | 去重结果可能误报（相似但不雷同的章节被标记） |

> **建议**：保留。去重校验的1次 LLM 调用成本远低于生成数章后才发现重复再重写。建议增加"自动应用建议"按钮，让用户一键采纳 LLM 的替换方案。

---

## 改动 3：章节目录每章显示可编辑摘要 + 定稿状态勾勾

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **每章显示** | 单行：编号 + 小图标 + 标题 | 卡片：编号 + 红绿勾勾 + 标题 + 大文本框 |
| **摘要** | hover 时显示 tooltip，不可编辑 | 大 textarea，用户可自由编辑修改 |
| **摘要来源** | 仅来自 blueprint 生成时的 summary | 未定稿→blueprint 摘要（可编辑）；定稿后→基于全文重新生成的摘要 |
| **定稿指示** | 小图标（check_circle / edit_note） | 红勾勾 ✗（未定稿）/ 绿勾勾 ✓（已定稿） |

| 优点 | 缺点 |
|:---|:---|
| 用户对每章剧情有完全控制权，生成前就能调整方向 | 500 章小说每章一个 textarea，页面极长 |
| 定稿后摘要自动更新，摘要更准确反映实际内容 | 用户手动修改的摘要和系统自动生成的摘要可能冲突 |
| 红绿勾勾直观，一眼看出哪些章节已定稿 | 需要在"定稿"步骤也能看到和编辑摘要，否则两个步骤脱节 |

> **建议**：采用折叠/展开模式——默认只显示标题行，点击展开才显示 textarea。另外，用户手动修改的摘要加"已手动修改"标记，后续系统更新时不覆盖。摘要编辑应在"章节目录"和"定稿"两个步骤都能操作。

## 改动 4：场景类型追踪（一致性审校新增）

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **meta 提取** | extractChapterMeta 输出：summary、timeline、伏笔 | 新增 `sceneTypes: string[]`（如：战斗升级、密谋揭露、感情升温） |
| **LLM 调用成本** | 每章1次（meta 提取） | 不增加调用，只在现有提取中多解析一个字段 |
| **审校维度** | 8 维 | 9 维，新增"场景多样性"检查 |

| 优点 | 缺点 |
|:---|:---|
| 几乎零成本——复用现有 meta 提取，不增加 LLM 调用 | LLM 提取的场景类型可能不够准确 |
| 直接告诉 LLM "最近用了什么场景"，比事后审校更前置 | LLM 被告知"避免重复"后仍可能不遵守 |
| 预设20种场景类型，覆盖大多数网文场景 | 固定列表不覆盖所有小说类型（如硬科幻、推理） |

> **建议**：保留，但增加两个改进：①根据用户选择的 genre 动态调整场景类型列表；②在 sceneTypes 之外增加 `pacingTag`（紧张/舒缓/过渡）和 `emotionIntensity`（高/中/低），这样能同时避免"连续3章高强度战斗"和"连续3章平淡日常"两种单调。

---

## 改动 5：草稿生成注入"近期已使用场景类型"

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **上下文注入** | 前章摘要、运行摘要、近10章摘要、前2章结尾、角色状态、伏笔、时间线 | 上述全部 + "近5章已使用场景类型" |
| **首章处理** | 与其他章节相同 | 首章不注入场景类型（无前文） |
| **重复警告** | 无 | 连续3章出现相同场景类型时加 ⚠️ 警告 |

| 优点 | 缺点 |
|:---|:---|
| LLM 生成时就能感知"最近一直在写战斗"，主动换场景 | prompt 变长，注入信息更多 |
| 首章豁免，逻辑合理 | 效果依赖 LLM 的指令遵循能力 |
| 与改动4配合，形成"提取→注入→避免"的闭环 | 如果场景类型提取不准确，错误信息会误导生成 |

> **建议**：保留。这是改动4的自然延伸，两者配合形成闭环。只需确保场景类型提取的准确率即可。

## 改动 6：强制过渡衔接

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **prompt 要求** | 一句"自然衔接前后章节" | 明确规则：场景/时间/情绪变化时必须交代过渡 |
| **审校检查** | 无专门检查 | 新增"过渡段质量"维度，检查前200字是否与上章结尾衔接 |

| 优点 | 缺点 |
|:---|:---|
| 解决"上一章在山洞、下一章突然在宫殿"的突兀跳转 | 硬性要求"前200字必须过渡"可能造成拖沓 |
| 审校有专门检查，执行有保障 | 有些章节故意用跳转制造悬念（如"与此同时，千里之外..."） |
| 消除读者最常感知到的"不连贯" | 快节奏小说（如连续战斗）每章都过渡会打断节奏 |

> **建议**：**不要硬性要求"前200字必须过渡"**。改为"**场景变化时必须过渡，场景连续时自然延续**"。具体 prompt 改为：如果本章开头的场景/时间/情绪与上一章结尾不同，必须在开头明确交代过渡；如果场景连续，自然延续即可。禁止无过渡的突兀跳转。

## 改动 7：runningSummary 结构化

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **格式** | 纯文本（200-500字），LLM 自由发挥 | 结构化 JSON，6个固定字段 |
| **内容** | 压缩摘要，重要细节可能在多次压缩中丢失 | `mainPlotProgress`：主线进展 |
|  |  | `activeConflicts`：活跃冲突列表 |
|  |  | `unresolvedMysteries`：未解之谜列表 |
|  |  | `emotionalTone`：当前情感基调 |
|  |  | `recentEvents`：最近3-5章关键事件 |
|  |  | `powerBalance`：势力格局状态 |
| **漂移风险** | 高——几十轮压缩后早期细节几乎必然丢失 | 低——结构化字段强制保留关键维度 |
| **注入方式** | 直接注入纯文本 | 格式化为分段可读文本，LLM 更易理解 |

| 优点 | 缺点 |
|:---|:---|
| **本次改造中收益最大的单项改进** | JSON 解析可能偶尔失败，需要 fallback 机制 |
| 纯文本摘要几十轮后几乎不可避免丢失信息，结构化强制保留 | 结构化摘要 token 量略大于纯文本 |
| 每个维度都有保证，不会因为"压缩"而遗漏关键线索 | 需要修改 updateRunningSummaryPrompt 和所有读取 runningSummary 的地方 |

> **建议**：**增加第7个字段 `keyTurningPoints: string[]`**——全书重大转折点列表，只增不减。即使写到第100章，第5章的重大转折仍然保留。这是解决"跨几十章连贯性"最有效的方式。另外 prompt 中要强调输出必须是合法 JSON，解析失败时 fallback 重新生成。

## 改动 8：多章全文上下文窗口

|  | 改 前 | 改 后 |
|:---:|:---|:---|
| **前章信息** | 前章最后 500 字 + 50-100 字摘要 | 前章**完整全文** + 前前章**完整全文** |
| **信息量** | 每章约 600 字前文信息 | 每章约 20000 字前文信息（增长 33 倍） |
| **跨章连续性** | 依赖摘要和 runningSummary（间接） | 直接看到前文全文（直接） |
| **token 消耗** | 每章生成输入约 1-2 万 token | 每章生成输入约 3-5 万 token（+2-3 万） |

| 优点 | 缺点 |
|:---|:---|
| **解决"剧情不连贯"最直接最有效的方法** | 每章生成 token 消耗增加约 2-3 万 |
| LLM 直接看到前文，不会出现"前文刚写过的细节后面又矛盾" | 弱模型（7B）可能因上下文过长导致注意力分散 |
| 不需要任何中间层压缩，信息零损失 | 全书蓝图、角色、伏笔等上下文叠加后可能达到 5-6 万 token |

> **建议**：token 无上限的前提下，这是最值得做的改进。建议根据章节类型动态调整注入量：普通章节注入前2章全文；关键转折章节（第1章、高潮、结局）注入前3章全文。配合结构化 runningSummary 中的 `keyTurningPoints` 字段，既能保证短距离连续性（前文全文），也能保证长距离连续性（转折点时间线）。

|:---|:---:|:---:|:---:|:---:|:---|
| 1.去卷数 | — | — | 无 | 中 | ✅ 保留 |
| 2.小说大纲+去重 | ⭐⭐⭐⭐ | — | +1次调用 | 中 | ✅ 保留 |
| 3.可编辑摘要+状态 | ⭐⭐ | ⭐⭐ | +1次调用/定稿 | 中 | ✅ 保留，加折叠 |
| 4.场景类型追踪 | ⭐⭐⭐ | — | 无 | 低 | ✅ 保留，加节奏标签 |
| 5.草稿注入场景 | ⭐⭐⭐ | — | 无 | 低 | ✅ 保留 |
| 6.过渡衔接 | — | ⭐⭐⭐⭐ | 无 | 低 | ⚠️ 改为"按需过渡" |
| 7.结构化摘要 | — | ⭐⭐⭐⭐⭐ | 无 | 中 | ✅ 保留，加转折点字段 |
| 8.全文上下文 | — | ⭐⭐⭐⭐⭐ | +2-3万token/章 | 低 | ✅ 保留，关键章加量 |

## 我的 3 处调整建议

| # | 原方案 | 调整为 | 理由 |
|:---:|:---|:---|:---|
| 1 | 前200字必须过渡 | **场景变化时必须过渡，连续场景自然延续** | 硬性200字过渡会造成拖沓，扼杀叙事灵活性 |
| 2 | 结构化摘要6个字段 | **增加 `keyTurningPoints` 只增不减的转折点列表** | 解决跨几十章连贯性的关键锚点 |
| 3 | 场景类型只有标签 | **增加 `pacingTag`（紧张/舒缓/过渡）和 `emotionIntensity`（高/中/低）** | 同时避免场景重复和节奏单调 |
